Accessibility as a Standard

Originally, web pages weren't much more than some text on a page with minor changes to color and position. However, as our websites became more enriched with different types of tags with their own style rules, they started to become less and less accessible. As we introduced style rules to completely change where an element appears or how it is displayed, and we began layering more tags to achieve complex designs, assistive technologies had more work to do to help users navigate the web.

When we say accessibility, we are referring to how well our site can be interacted with by users with disabilities. Much like our discussion of responsive web design last week, accessibility covers a wide range of different user experiences. Users may have vision problems, such as reduced visual clarity or color perception. They may even be completely blind, and use text-to-speech screen readers to navigate the web. They may also have motor problems and use input methods other than a typical keyboard and mouse.

As assistive technologies evolved to meet the new enriched web, we came across patterns of websites that could effectively be translated to other modalities, and others which were hard to navigate. To help codify these patterns for web developers, we created the Accessible Rich Internet Applications (ARIA) standard.

The Five Rules of ARIA

The core of the ARIA standard is five rules that guide the development of websites. Those rules are:

  1. When possible, express content using meaningful semantic HTML Tags

    Screen readers specialize in figuring out how to display web content. If you use meaningful tags for your content, screen readers can often figure out how to appropriately present that content without any further intervention on your part!

  2. At the very least, do not use tags in a way that contradicts the normal semantic meaning.

    For the same reason that using tags semantically causes them to display appropriately on a screen reader, using tags in a way that conflicts with their normal meaning will cause assistive technology to incorrectly assume the role of an element. For example, turning a "header" tag into a clickable tab might cause confusion to users who are primarily hearing about tags by role. In particular, try not to cross the "interactive"/"non-interactive" barrier.

  3. Your website must be navigable without the use of a mouse.

    Mice are notoriously precise, and difficult to emulate if a user cannot actually manually point one. Instead, most users with assistive technology make use of tab-controls or another tool that uses a tab control (such as saying the word "next".) You should be able to navigate between controls and between text in a sane order.

  4. Do not hide elements that can trap focus.

    Since we are saying all controls should be easy to navigate between, it stands to reason that we should not "hide" controls that can still steal focus (since using tab may cause the site to behave differently than the way it does to normal users.) One exception to this: if you use display: none or visibility: hidden, an element can no longer be the target of focus, and its safe to hide it from the aria view.

  5. Controls on a site should have accessible and meaningful labels.

    Without visual cues, there is no direct link between labeling elements and the things they label: We must ensure that, if labels are contained in an element separate from the control themselves, that these labels are appropriately linked through the use of aria attributes.

Basic Considerations

"Click Me!" Links and Buttons

When we have a link, ideally we should put descriptive text in the link itself. When we can't do that, we can use the aria-label="" attribute to set additional text that can be associated with a link or button. We can add labels to our link and navigation drawer.

Hidden and Role Manipulation

When we show or hide controls, we should make those controls are actually added or removed from the document. For instance, we may want to ensure our navigation menu can only be tabbed to while it is on screen. Similarly, we can manipulate roles of tags to change how a screen reader reads them to be more in-line with a way a user would think about them. Watch what happens if we change the role of our checkbox to be a 'button'. There is one special role in ARIA: presentation. Watch what happens when we try setting the presentation role on an element. When we set a presentation role, those elements will be treated as plain text. We should be careful not to set important interactive elements to the presentation role.

Controlling Tab Order

We can set a tabindex="" on each focusable element to specify the order we want to tab through them, starting with one. A tabindex will even allow things we can normally not tab to steal focus on the page.

Controlling Text Read Order

In addition to tab order, we can also set normal text order using the aria-flowto property. This property specifies the next thing we want to flow to when using normal text navigation in a screen reader.
