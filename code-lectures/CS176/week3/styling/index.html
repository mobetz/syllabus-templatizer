Before we get started talking about how to style our pages, let's learn a bit of background about how the computer thinks about a web page.

The Document Object Model

We can represent the content of a web page in many different formats.
  • Last class, we learned how to represent web content as nested tags using HTML.
  • The browser represents web content as a bunch of elements positioned across a page.
  • Programming languages (like Javascript) need to represent web content as classes and objects.

It turns out, we can notate all of these different representations in a single, platform-agnostic format, which we call the Document Object Model. The Document Object Model represents all of the nested sets of tags that we learned about last class as nodes into a tree (sort of like the trees you see in the left pane of a file browser when expanding a series of folders.)

In the DOM, each 'level' on the tree represents a tag that surrounds all of its children nodes. When the browser receives HTML code, the first thing it does is convert that code into a DOM tree like the one pictured here.

Text is really inconvenient to work with as a computer... every time you load the page, it needs to go out and find where tags start and end, what 'sections' of the content exist to layout. Programs are really good at working with tree-like data, and we've developed many techniques for searching and manipulating trees efficiently. As a consequence, converting an HTML document into a DOM tree allows the browser to easily navigate the page when applying styling and interacting with JavaScript code.

Block and Inline Elements

When the browser wants to convert a DOM tree into a rendered website, one of the first things it has to do is identify the major "blocks" of content and calculate where each should be positioned on the page.

It does this by scanning all the nodes in the DOM, and figuring out which ones actually represent new blocks of content, and which only change something that exists inside a block.

It turns out, we can pretty cleanly divide (almost) all HTML tags into two categories:

Block elements represent a new block of content. When the browser renders them, it assumes they're going to take up the entire width of the page, and gives them a blank line of space above and below unless otherwise specified.

Inline elements represent labels, styling, or elements that otherwise "fit" inside an existing block, but don't carve out their own space.

So why do we care about all of this? Because it's going to inform how we can add our own additional styles to our web pages using another language called CSS!

What is CSS?

CSS stands for "Cascading Style Sheets". Cascading Style Sheets are an extension to HTML that allows us to declare special "rules" that modify our HTML tags. Like HTML itself, these rules are 'declarative' -- a CSS stylesheet is a list of information, and the browser is responsible for interpreting that information to produce a rendered web page.

The most common place to put these rules is in a separate file, which we can 'attach' to this document using a <link> tag. It contains a rel attribute which indicates that this link is to a stylesheet, and it contains an href attribute similar to our anchor tags, that indicates where this stylesheet is located.

CSS rules follow a special syntax, similar to our tags:

  • First, they specify a 'selector' or group of selectors, which represents what elements should be changed.
  • Next, they open a 'block' of declarations using a pair of {curly braces}.
  • Inside this block, they define one or more property:value pairs separated by a colon, with a semicolon at the end.

What is it that makes Cascading Style Sheets "cascading"? -- We can apply multiple rules that each overlap. However, when these rules conflict, one has to win. When this happens, the most specific rule wins:

  • Style added 'inline' with a style attribute is more specific than ID.
  • IDs are more specific than classes.
  • Classes are more specific than elements.
  • Tags that are 'inside' other tags are more specific than their containers.
  • When two rules 'tie', the last rule applied wins.

Selectors

A selector is the first part of our CSS style rules. A selector lets us specify the element or elements that we want to modify with our properties. Selectors are powerful, and contain many flourishes that let us precisely point to specific elements or groups of elements. We will be discussing a few of these next class.

However, the simplest selectors are just tags! If we provide the name of the tag in the selector region of a CSS rule, then that block of declarations will apply to all matching tags in our document. Let's try this out in our common_styles.css by setting default styling for our sections, images, and body.

In addition, there are two other common selectors: id selectors and class selectors. Every single tag in HTML has two attributes in common -- we can specify an id that uniquely identifies that specific element, and a class that labels that element as being part of a set of similarly styled groups (each group is separated by a space). Then, in our CSS rules, we can use #ID_NAME to refer to an element by id, and .CLASS_NAME to refer to an element by class.

Common Properties

Colors

There are several declarations that deal with colors.
  • color sets the color of text.
  • background-color sets the color for the whitespace behind the element.
  • border-color sets an optional outline for a block.
All of these color-based properties use 'web colors'. Web colors can be specified in a few ways: either by using a six digit hexadecimal number, a three digit hexidecimal number, a special color name, or the rgb() or hsl() css "functions". When we use a three or six digit number, each third of the number refers to a different color: the first third holds information about red values, the next third holds information about green values, and the final third holds information about the blue values:

Length

Length is used for properties such as width, height, margin, and padding. Length can be specified as:
  • pixels (px)
  • % of screen (%)
  • "size of the letter m"-s (em)
  • and so many more...

An element's width and height aren't always its width and height. The content area of an element (where text and child elements will actually display) does not include its margin (the extra space left beyond the edge of the element), border (a styled line around the edge of the element), or padding (space inside the element's "box",) by default. These different measures of space around an element are collectively referred to as the "box model".

However, if we use the "box-sizing: border-box" attribute for an element, then padding and border are included in the width. Where possible, you should try to position elements using only these tools we've learned about today: (width, height, margin, and padding.) Doing this will help your page reflow properly on screens of different sizes. Next class we'll learn a few tricks for when that's insufficient.
